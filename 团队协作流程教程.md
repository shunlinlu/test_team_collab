# team-collab 协作流程上手教程

给团队成员照着熟悉整套流程用。读完 + 跟着"动手练习"走一遍,就能上手协作。
本项目(test-team-collab)本身就是练习用的,放心操作。

---

## 1. 一句话心智模型

团队协作 = 两个 git 仓 + 一块共享"项目大脑" + 两个同步命令。

```
┌─ 代码仓 (test_team_collab_code) ──────── 放真正的代码,改动走 PR
│
└─ 文档仓 (test_team_collab) ── clone 进代码仓的 obsidian-docs/
     └─ 状态四件套 = 共享项目大脑
        CURRENT.md  现在什么状态
        NEXT.md     接下来做什么
        RISKS.md    有什么坑/风险
        TODO.md     有哪些活、谁在做 (@owner)
     └─ _handoffs/       每次 session 的交接记录(只增不改)
     └─ dev-records/<你>/ 个人开发笔记
```

- **代码**走代码仓;**"谁在干嘛、干到哪、下一步"**走文档仓。两条线分开。
- 同步靠两个命令:`$checkpoint`(本地存草稿) 和 `$handoff`(发布并推送)。

---

## 2. 一次性:加入项目 (join)

每个人在自己机器上做一次。

**前置:** 装好 `team-collab` CLI,配好能推 GitHub 的 SSH key,设好 git 身份
(`git config --global user.name/user.email`)。

```bash
# clone 代码仓,再把文档仓 clone 进它的 obsidian-docs/
git clone git@github.com:shunlinlu/test_team_collab_code.git test_team_collab
cd test_team_collab
git clone git@github.com:shunlinlu/test_team_collab.git obsidian-docs

# 登记到本机(换成你自己的用户名)
team-collab init --join test-team-collab \
  --code "$PWD" --docs "$PWD/obsidian-docs" \
  --user <你的用户名> --code-platform github
```

加入后先读 `obsidian-docs/CURRENT.md` 和 `OVERVIEW.md`,建立全局认知。

## 3. 日常一天的完整流程

```
开工
 ├─ 1. 拉最新:两个仓都 git pull（docs 仓用 git pull --rebase）
 ├─ 2. 读 CURRENT → NEXT → TODO，搞清现状和该干什么
 ├─ 3. 从 TODO 挑一条，加上 @你的用户名 认领，挪到「进行中」
 ├─ 4. 在代码仓干活；改动 commit 到代码仓（后续走 PR）
 ├─ 5.（中途）想存一下当前状态 → 说 $checkpoint（只改本地，不推）
 └─ 6.（收工）→ 说 $handoff <本次主题>
        Claude 会：pull → 写一份 _handoffs 交接 → 更新四件套
                 → commit → rebase → push 到 docs 仓
收工
```

同事第二天 `git pull`,就能从你的 handoff + 更新后的四件套无缝接手。

## 4. 共享状态怎么写(四件套规则)

核心原则:**简洁、结构化、靠链接**。四件套是"摘要缓存",不是日志。
长篇背景 / 实验细节 / PR 证据放到 `_handoffs/`、`dev-records/`、`archive/` 或设计文档,
这里只留一句话 + 一个 Markdown 链接。

| 文件 | 写什么 | 别写什么 |
|---|---|---|
| CURRENT.md | 一句话状态、当前焦点、3–5 条里程碑、活跃分支 | commit 列表、长时间线 |
| NEXT.md | 下一步动作(**动词开头** + 验收标准/链接)、待决策 | 模糊的"搞一下 X" |
| RISKS.md | 风险表:风险/影响/owner/状态/缓解 | 没有 owner 的风险 |
| TODO.md | 每条一个可认领的活,带 `@owner` | 一大段背景(挪去 NEXT) |

**校验命令:**

```bash
team-collab lint-state obsidian-docs   # 只查四件套
team-collab lint       obsidian-docs   # 全量(含 TODO 所有权、frontmatter)
```

> 踩坑提醒:NEXT.md 每条必须**同时**含 ①白名单动词(run/add/update/fix/verify/
> test/create/review/ship/deploy… )②验收标准或 `[链接]()`。否则 lint 会警告。
> 例:`Run \`brew install gitleaks\`; acceptance: pre-commit 真正扫描 (见 [RISKS](./RISKS.md))`。

## 5. TODO 的 `@owner` 认领机制(两人协作关键)

```markdown
## 进行中
- [ ] 实现数据清洗模块 @alice since 2026-07-02 (id: T-20260702-01)
```

- `@owner` 是**公开的意图信号**:别人看到就知道这条有人做了,别抢。
- 认领 = 加上 `@你`、挪到「进行中」、写日期。完成 = 打 `[x]` 挪到「最近完成」。
- 它**不是强锁**,是"乐观并发":真撞车了靠 git 冲突兜底,谁后 push 谁负责合。
- **不要动别人 `@owner` 的条目**(除非当面说好)。
- 复杂时保留 `id: T-YYYYMMDD-NN` 稳定编号,方便对齐。

## 6. `$checkpoint` vs `$handoff`

区别只有一个:**动不动 git**。

| | `$checkpoint` | `$handoff <主题>` |
|---|---|---|
| 时机 | session 中途 | session 收尾 |
| 动作 | 只更新四件套本地文件 | pull → 写交接 → 更新四件套 → commit → rebase → push |
| git 操作 | **零** | 完整 |
| 同事看得到吗 | 看不到(没推) | 看得到(推了) |
| 频率 | 一个 session 可多次 | 收尾一次 |

一句话:**checkpoint 是存草稿,handoff 是发布给对方看。**

`$handoff` 还有个"空 session 保护":这次没干实事它会拒绝伪造交接。

## 7. 安全红线(务必遵守)

- 文档仓**永远不要 `git add .`**,只精确 add 你要提交的文件。
- **永远不要 `git commit --no-verify`**,不要绕过 gitleaks 密钥扫描。
- 不要 force-push 受保护/共享分支(`main` 等)。
- 遇到 pull 冲突、push 被拒、钩子失败 → **停下报告**,不要盲目"强行恢复"。
- 别往文档里写密钥、密码、客户隐私、未公开的内部信息。

## 8. 常用命令速查

```bash
team-collab doctor --project test-team-collab   # 体检:工具/wiring 是否正常
team-collab health --docs obsidian-docs         # 有没有没推上去的改动
team-collab lint       obsidian-docs            # 文档质量全量检查
team-collab lint-state obsidian-docs            # 只查四件套
$checkpoint                                     # 中途快照(对 Claude 说)
$handoff <主题>                                  # 收尾交接(对 Claude 说)
```

## 9. 动手练习(建议每人跑一遍)

1. 按第 2 节 join 本项目,`git pull` 两个仓。
2. 读 `CURRENT.md` / `OVERVIEW.md`,说出这个项目现在什么状态。
3. 在 `TODO.md` 挑一条 backlog,加 `@你` 认领,挪到「进行中」。
4. 在代码仓做个小改动(比如给 `collector/collect.py` 加个字段或注释),
   `python3 -m unittest discover -s tests` 跑通,commit。
5. 说 `$checkpoint`,看四件套本地变化(还没推)。
6. 说 `$handoff 练习-你的名字`,看它生成 `_handoffs/` 交接并 push。
7. 让同事 `git pull`,确认他能看到你的 handoff 和更新。

跑通这 7 步,你就完整体验了一遍协作闭环。
